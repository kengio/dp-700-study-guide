# DP-700 Hands-on Labs

A ten-lab pack that builds one working Fabric solution end to end, in the order a real data engineering project would: stand up the workspace, wire lifecycle and security around it, orchestrate the flow, then ingest, load, transform, warehouse, stream, and finally monitor and optimize. Every lab chains onto a single shared workspace and a small synthetic retail dataset generated in Lab 01 — no external accounts or downloads required for the core data path; Lab 02 does need a free GitHub or Azure DevOps account for Git integration.

> [!abstract]
>
> - Ten labs covering all three DP-700 domains hands-on: workspace/capacity setup, lifecycle & Git, security & DDM, orchestration, batch ingestion, loading patterns, PySpark transformation, warehouse T-SQL, streaming, and monitoring/optimization
> - All labs chain on **one workspace** (`dp700-labs`) and **one synthetic dataset** (customers/products/orders/order_items/events) generated by a PySpark cell in Lab 01 — nothing to download
> - Each lab lists exactly what to keep for later labs in its **Cleanup** section — skipping cleanup guidance can break a downstream lab
> - Requires only a free 60-day Fabric trial; a handful of steps that need an external cloud account (ADLS/S3, Azure SQL Database) are marked optional with a documented skip path

> [!tip] What the Exam Tests
>
> - DP-700 is a hands-on exam about a portal and a set of code surfaces (PySpark, T-SQL, KQL) — reading the topic files alone under-prepares you for "which button, which parameter, which exact syntax" questions
> - These labs build the muscle memory the multiple-choice questions assume you already have: creating a custom Spark pool, writing a `CREATE SECURITY POLICY`, wiring a `ForEach` around a `Notebook` activity, reading `$.result` from a masked column
> - Hands-on practice surfaces gotchas pure reading misses — a Commit button greyed out because of pending Git updates, `ALTER FUNCTION` blocked by an active security policy, `ForEach` refusing to nest

## Setting Up Your Fabric Trial

Every lab in this pack runs on the free, 60-day Microsoft Fabric trial capacity — no Azure subscription or credit card required to start. Steps below are summarized from the official trial documentation; if the portal has moved since this was written, follow the linked doc instead of guessing at a stale click path.

> [!warning] Prerequisite: Work or School Account Required
> The Fabric trial requires a **work/school (Microsoft Entra ID) account** — a personal Microsoft account (outlook.com, hotmail.com, or an MSA tied to a personal Gmail sign-in) is **not eligible** and can't start a trial. If you don't have a work or school account, the [Microsoft 365 Developer Program](https://learn.microsoft.com/en-us/office/developer-program/microsoft-365-developer-program) is the documented path to a free developer tenant with its own Entra ID directory — verified live as of this writing. Note that automatic qualification for the program's free Microsoft 365 E5 sandbox subscription now favors Visual Studio subscribers, partners, and support-contract holders; an unaffiliated individual signup may see reduced eligibility, so check the program's current criteria before treating it as a guaranteed path.

1. **Sign in** to [https://app.fabric.microsoft.com](https://app.fabric.microsoft.com) with a Microsoft 365 / Entra ID account (a personal Microsoft account will not work — see the prerequisite above). If your organization has never enabled Fabric, you may need a Fabric (Free) license first — the portal prompts for this automatically the first time you try to use a Fabric feature.
2. **Start the trial**: select your account photo in the top-right corner → **Account manager** → **Start trial**. If you don't see a **Start trial** option, your tenant admin may have disabled trials — see the [Fabric trial capacity docs](https://learn.microsoft.com/en-us/fabric/fundamentals/fabric-trial) for the alternate "trigger a trial by using a feature" path.
3. **Activate**: in the **Activate your 60-day free Fabric trial** dialog, review the **Trial capacity region** dropdown (default is usually your tenant's home region — keep it unless you have a specific reason to change it, since moving a populated workspace to a different region later requires deleting its items first), agree to the terms, and select **Activate**.
4. You're now the **Capacity administrator** of an F4 (or F64, depending on tenant eligibility) trial capacity, plus a complementary Power BI Individual trial. The trial runs for **60 days**; check remaining days any time via **Account manager → Trial status**, or **Admin portal → Capacity settings → Trial**.

> [!warning] Common Mistake
> Don't confuse the trial capacity (analytics compute + 1 TB OneLake storage, what these labs need) with a plain Power BI Individual trial (reporting only, no Fabric items). If workspace creation later fails to offer a **Fabric Trial** or **Trial** workspace type, the Fabric trial itself didn't activate — retry step 2–3 before continuing.

### Creating the Lab Workspace and Assigning Capacity

1. In the Fabric portal nav pane, select **Workspaces → New workspace**.
2. Name it **`dp700-labs`** — every lab in this pack assumes this exact name so relative paths and cross-item references (Lakehouse names, shortcut targets) match what's documented.
3. Expand **Advanced** → **License mode** (or **Workspace type**, depending on portal version) → select **Trial** (or **Fabric Trial**) → confirm the trial capacity you activated above is selected.
4. Select **Apply**. The empty workspace is ready — Lab 01 populates it.

If you created the workspace before activating the trial, you can retarget it any time: **Workspace settings → Workspace type/License → Edit → Trial**. Full details: [Create a workspace](https://learn.microsoft.com/en-us/fabric/fundamentals/create-workspaces).

## The Shared Lab Dataset Convention

Every lab in this pack — 01 through 10 — operates on **one workspace** and **one small synthetic retail dataset**, so nothing after Lab 01 requires downloading a file or provisioning an external data source for the core path.

- **Workspace**: `dp700-labs`, created above. Labs never create a second permanent workspace except where a lab topic *requires* one (e.g., Lab 02's deployment-pipeline Test stage, which is explicitly optional to keep afterward).
- **Dataset**: generated entirely in-notebook by a single PySpark cell in [Lab 01](./01-workspace-capacity-setup.md#step-7-generate-the-shared-retail-dataset), landing in a lakehouse named **`lh_bronze`**:

  | Table | Rows | Purpose |
  | :--- | :--- | :--- |
  | `customers` | 50 | Dimension table; includes a synthetic `ssn` column used for DDM/CLS masking exercises in Lab 03 |
  | `products` | 30 | Dimension table |
  | `orders` | 200 | Fact table (header) |
  | `order_items` | ~500 | Fact table (line items), foreign keys to `orders` and `products` |
  | `events` | 500 | Clickstream-style event log (`page_view`/`add_to_cart`/`purchase`/`search`), used by Lab 09's Eventstream/KQL lab |

  All five land as **managed Delta tables** (`Tables/`) for immediate SQL analytics endpoint and Spark access, plus a CSV copy under `Files/raw/` so later labs have a file-based source to practice Copy activity and shortcuts against without needing an external account.
- **No external downloads**: the dataset is entirely synthetic and generated with Python's `random` module inside the notebook cell — there's nothing to fetch, no license to check, and it reproduces identically (seeded) if you need to regenerate it.
- **Later labs add, they don't replace**: Lab 05 adds a second lakehouse (`lh_silver`); Lab 08 adds a warehouse (`wh_gold`, first created in Lab 03); Lab 09 adds an Eventstream and a KQL database. All of them read from or extend the Lab 01 dataset rather than generating a new one.

> [!note] Mental model — one dataset, ten labs
> Think of Lab 01 as pouring the foundation slab once. Every subsequent lab builds a different room on the same foundation — nobody re-pours concrete. If a later lab's step fails because a table or lakehouse from an earlier lab is missing, the fix is almost always "go back and complete that earlier lab's Cleanup section," not "regenerate the dataset."

## If You Only Have Time for 3

Do these, in order:

1. **[Lab 01 — Workspace & Capacity Setup](./01-workspace-capacity-setup.md)** — not optional. Every other lab depends on the workspace, lakehouse, and dataset it creates.
2. **[Lab 04 — Orchestration: Pipelines & Notebooks](./04-orchestration-pipelines-notebooks.md)** — the pipeline canvas (`Lookup` → `ForEach` → parameterized `Notebook`, on-failure branches, dynamic expressions) is Domain 1's most hands-on surface and the one candidates most often study only from screenshots. Building one for real fixes that.
3. **[Lab 09 — Streaming: Eventstream & KQL](./09-streaming-eventstream-kql.md)** — the surface furthest from typical batch-engineering muscle memory, spanning Eventstream configuration and KQL syntax most candidates have never typed by hand before this exam.

This trio touches all three domains and both of the exam's most-tested "type the exact syntax" surfaces (pipeline expressions and KQL). If a fourth slot opens up, add [Lab 03 — Security, OneLake & DDM](./03-security-onelake-ddm.md) for the RLS/CLS/DDM T-SQL practice — it's the single highest-density topic for exam traps.

This trio is optimized for syntax-surface coverage, not raw blueprint weight — if you're maximizing strictly against the July 2026 blueprint's domain percentages, swap Lab 09 for Lab 03 (Security's RLS/CLS/DDM density scores higher against Domain 1's weighting than streaming scores against Domain 2), or, time permitting, add [Lab 10 — Monitor & Optimize](./10-monitor-optimize.md) as a fourth pick instead of trading anything away.
