# Governance

## Overview

Fabric governance spans three related but distinct mechanisms: **sensitivity labels** classify and protect content (via Microsoft Purview Information Protection), **endorsement** (Promoted/Certified/Master data) signals content quality and trustworthiness, and the **unified audit log** (via Microsoft Purview) records who did what to which item. This topic covers all three, verified against Microsoft Learn as of **July 2026**.

> [!abstract]
>
> - **Sensitivity labels** apply to Fabric/Power BI items, can inherit downstream (Fabric item → Fabric item, Fabric item → Power BI item, but **not** Power BI item → Fabric item), and support both tenant-wide and domain-level default label policies
> - **Endorsement** has three tiers: **Promoted** (any item owner/write-permission holder), **Certified** (only tenant-admin-authorized reviewers), and **Master data** (also only authorized reviewers) — each requires write permission on the item as a baseline
> - Fabric activity is recorded in the **unified audit log**, accessed via the Microsoft Purview portal, searchable via UI filters or the `Search-UnifiedAuditLog` PowerShell cmdlet
> - Audit log retention defaults to **180 days** for non-E5 users, **1 year** for E5-licensed users (Entra ID/Exchange/SharePoint always get 1 year by default) — up to 10 years with the retention add-on

> [!tip] What the Exam Tests
>
> - Which endorsement tier a given user can apply, based on their permission level and whether they're an authorized reviewer
> - Sensitivity label downstream inheritance rules — what propagates and what explicitly does not (Power BI → Fabric direction)
> - Domain-level vs. tenant-level default sensitivity label policies, and which one wins when both exist
> - Where Fabric audit activity is viewed/searched, and default retention periods by license tier

## Sensitivity Labels

Sensitivity labels come from **Microsoft Purview Information Protection** and apply to Fabric and Power BI items to classify and, where a protection policy is attached, restrict access to sensitive content.

### Applying Labels

Labels can be applied:

- **Manually** — by any user included in the tenant's list of users authorized to apply labels (labels must also be *published* to that user via Purview label policy)
- **By default** — a default label policy applies automatically to an item when created/edited, if the user doesn't choose one explicitly
- **Programmatically** — via Power BI admin REST APIs

### Downstream Inheritance

When a label is applied to an item, it propagates **downstream** to dependent items by default:

| Direction | Supported? |
| :--- | :---: |
| Power BI item → Power BI item | ✅ |
| Fabric item → Fabric item | ✅ |
| Fabric item → Power BI item | ✅ |
| ==Power BI item → Fabric item== | ❌ Not supported |

> [!warning] Common Mistake
> Sensitivity label inheritance is **one-directional between Power BI and Fabric**: a Fabric item's label flows to a downstream Power BI report or semantic model, but a Power BI item's label does **not** flow back upstream into a Fabric item. A scenario describing "a labeled semantic model automatically labels the Lakehouse it was built from" is describing something that isn't supported.

Autogenerated items from a Lakehouse or Warehouse (e.g., its SQL analytics endpoint) take their label from the **immediate parent**, not further upstream. **Inheritance upon creation** — a *new* item created from an existing one inherits that item's label — works for all Power BI items and specific non-Power BI patterns (a Pipeline or Notebook created from a Lakehouse, a shortcut created from a Lakehouse, a Queryset/Pipeline created from a KQL Database).

**Inheritance from data sources** (a Fabric item ingesting from a source that already carries a label) is currently supported **only for Power BI semantic models** — it doesn't extend to Lakehouse/Warehouse ingestion from a labeled source.

### Export

Sensitivity labels (and any attached protection) travel with content through specific **supported export paths**: export to Excel/PDF/PowerPoint, Analyze in Excel, PivotTable in Excel (M365 E3+), and download to Power BI Desktop (`.pbix`). Export to `.csv`/`.txt`, and any Fabric experience export not on this list, does **not** carry the label — though a warning is shown if the exported item had one.

### Labeling Modes and Licensing

Fabric's information protection capability table distinguishes four labeling modes, each with its own maturity:

| Mode | Behavior | Support |
| :--- | :--- | :--- |
| Manual labeling | A user on the tenant's authorized-labeler list applies a label directly | Supported for all Fabric items |
| Default labeling | An unlabeled item gets a default label automatically on creation/edit | Supported for all items, with limitations on which UI paths trigger it |
| Mandatory labeling | An item can't be saved without a label | Fully supported for Power BI items only; partial support elsewhere |
| Programmatic labeling | Labels set/changed/removed via Power BI admin REST APIs | Supported for all Fabric items |

Applying labels to Power BI items also requires a **Power BI Pro or Premium Per-User (PPU)** license, in addition to whatever Microsoft Purview Information Protection licensing the tenant uses. Sensitivity labels in Power BI Desktop require the **December 2020 release or later** — an older Desktop build fails outright when opening a protected `.pbix` file rather than silently ignoring the label.

> [!note]
> For **mandatory labeling** on non-Power BI Fabric items (Lakehouses, pipelines, Warehouses), the enforcement is currently soft: if mandatory labeling is on but default labeling is off, a user *can* select a label in the UI, but nothing actually blocks saving the item unlabeled. Don't assume mandatory labeling behaves identically for Fabric items and Power BI items — only the Power BI side is a hard block today.

### Access Control via Sensitivity Labels

Sensitivity labels can restrict access outright — not just classify content — but only in three specific scenarios, each tied to a Microsoft Purview policy type:

| Scenario | Policy type | Effect |
| :--- | :--- | :--- |
| Accessing an item **in the tenant where the label was applied** | Protection policy | Access governed by the protection policy attached to the label |
| Opening a **Power BI Desktop (`.pbix`) file** with a protection-linked label | Publishing policy | Access depends on the opener's permissions under that publishing policy |
| Opening a file from a **supported export path** carrying a labeled item's protection | Publishing policy | Same publishing-policy-driven access check as the `.pbix` case |

All other scenarios — including **cross-tenant access** (e.g., external data sharing) and any export path not on the supported list (`.csv`, `.txt`) — do **not** get access control from the label, even if the label itself is still visibly attached.

### Domain vs. Tenant Default Labels

A tenant-wide default label policy can be **delegated to the domain level** if the *domain-level default sensitivity label* feature is enabled by the tenant admin. When enabled, a domain admin (or Fabric admin) sets **Delegated Settings → Information protection → Set a default label for this domain**, and that label applies by default to items in workspaces assigned to the domain — overriding the tenant-wide default for that domain's scope.

> [!note] Mental model — Domain defaults as a local dress code
> Think of the tenant-wide default label as the building's general dress code, and a domain-level default as one floor's stricter dress code for a specific department. Anyone walking onto that floor follows the floor's rule, not the building's — but the building's rule still applies everywhere the floor hasn't overridden it. The floor can only set a stricter or different local rule if building management (the tenant admin) has explicitly allowed floor-level overrides in the first place (feature must be enabled).

**Practice Question 1** *(Medium)*

A Lakehouse has no sensitivity label applied. A Power BI report is built from that Lakehouse's SQL analytics endpoint and manually labeled "Highly Confidential." Does the Lakehouse automatically inherit that label?

A. Yes — labels always sync bidirectionally between Fabric and Power BI items
B. No — downstream inheritance flows Fabric → Power BI, never the reverse
C. Yes, but only if the Lakehouse has default labeling enabled
D. No — inheritance only works for semantic models, never Lakehouses

> [!success]- Answer
> **B. No — downstream inheritance flows Fabric → Power BI, never the reverse**  
> Fabric documentation explicitly lists "Power BI item to Fabric item" as **not supported** for downstream inheritance. A label applied to the Power BI report has no effect on the upstream Lakehouse; if the Lakehouse needs a label, it must be applied there directly (manually, by default policy, or inherited from further upstream within Fabric).

**Practice Question 2** *(Medium)*

An organization migrated from legacy Azure Information Protection (AIP) labels years ago but never completed the move to the Microsoft Purview Information Protection unified labeling platform. What's the effect on using those labels in Fabric?

A. No effect — Fabric supports both AIP and Purview unified labels interchangeably
B. AIP labels must first be migrated to the Purview unified labeling platform before they can be used in Fabric
C. Fabric automatically converts AIP labels the first time they're applied to an item
D. AIP labels work in Power BI Desktop but not in the Fabric portal

> [!success]- Answer
> **B. AIP labels must first be migrated to the Purview unified labeling platform before they can be used in Fabric**  
> Fabric's information protection capabilities depend on Microsoft Purview Information Protection specifically. Organizations still on legacy Azure Information Protection labels must complete the migration to unified labeling before those labels are usable for classification and protection inside Fabric and Power BI — there's no interchangeable dual-support or automatic just-in-time conversion.

## Endorsement

Fabric offers three endorsement mechanisms to increase an item's visibility and signal trust:

| Tier | Meaning | Who can apply |
| :--- | :--- | :--- |
| ==Promoted== | "Worth using" — encourages collaborative reuse | Any item owner, or anyone with **write permission** on the item |
| ==Certified== | Meets the organization's quality/reliability bar | Only reviewers **authorized by the Fabric admin**, and only with write permission on the item |
| ==Master data== | Core, single-source-of-truth data (e.g., customer lists, product codes) — data items only | Only reviewers **authorized by the Fabric admin**, and only with write permission on the item |

All Fabric and Power BI items can be Promoted or Certified **except Power BI dashboards**. The **Master data** badge is restricted to items that actually contain data (Lakehouses, semantic models, and similar) — it doesn't apply to items like reports or pipelines.

If an item owner wants Certified or Master data status but isn't an authorized reviewer themselves, they must **request** it through their organization's process — the button appears greyed out in the item settings, with a link the Fabric admin configures to point at internal request instructions.

> [!warning] Common Mistake
> Don't conflate "who can *apply* an endorsement" with "who can *see* it applied." Any write-permission holder can **Promote**, but **Certified** and **Master data** are gated by tenant-admin-designated authorization — a scenario where a report owner tries to self-certify their own report without being on the authorized-reviewers list will show the Certified option greyed out, not silently succeed.

**Practice Question 3** *(Easy)*

A data engineer with write permission on a Lakehouse wants to mark it as **Certified**. They are not on the Fabric admin's list of authorized certifiers. What happens when they try?

A. The Certified option applies successfully since they have write permission
B. The Certified option is greyed out — they must request certification through their org's process
C. The item is automatically promoted instead
D. The Fabric admin is notified and auto-approves after 24 hours

> [!success]- Answer
> **B. The Certified option is greyed out — they must request certification through their org's process**  
> Write permission alone is necessary but not sufficient for Certified (or Master data) status — the user must also be on the Fabric admin's authorized-reviewers list. Without that authorization, the UI greys out the option and directs them to a request workflow the admin has configured.

## Fabric Audit Logs

Fabric activity — item creation/deletion, sharing changes, sensitivity label changes, domain administration, and more — is recorded in Microsoft 365's **unified audit log**, accessed through the **Microsoft Purview portal** (`https://purview.microsoft.com/audit`).

### Accessing and Searching

Requires the **Audit Logs** or **View-Only Audit Logs** role (via Exchange admin center role assignment, or a custom Purview role group). Search supports filtering by:

- Date/time range (UTC; default last 7 days; max 180 days per search)
- Activities (friendly names, grouped) or exact operation names
- Users (blank = all users and service accounts)
- File, folder, or site
- Record type (e.g., filter to sensitivity-label-related record types like `MIPLabel`)

```powershell
# Query the audit log programmatically (Exchange Online PowerShell)
Search-UnifiedAuditLog -StartDate (Get-Date).AddDays(-7) -EndDate (Get-Date) `
    -RecordType PowerBI -UserIds analyst@contoso.com
```

> [!note]
> To see **Power BI/Fabric-specific** activities in audit search results, auditing must first be enabled in the **Power BI admin portal → Audit logs** setting — it isn't automatically captured just because unified audit logging is on for the tenant.

### Administrative Units and Scoped Access

Access to search the audit log is scoped by the searching admin's assigned **administrative units** in the Microsoft Purview portal:

| Admin type | Scope available |
| :--- | :--- |
| Unrestricted admin (no admin units assigned — the default) | All activity logs, from any user, non-user, or system account |
| Restricted admin (one or more admin units assigned) | Only activity logs from users matching an assigned administrative unit |

A handful of audit activities (Azure Information Protection `Discover`, several Exchange/Dynamics 365 operations, endpoint DLP file events) are visible **only to unrestricted admins**, regardless of scoped-search permissions — a restricted admin's search simply omits them.

When capacity ID/name isn't available directly in an audit record, cross-reference it against the **Microsoft Fabric Capacity Metrics app** instead of assuming the data is missing entirely.

### Retention

| License / record type | Default retention |
| :--- | :--- |
| Entra ID, Exchange, SharePoint activity (any license) | 1 year (default policy, non-editable) |
| All other activity — Office 365/Microsoft 365 **E5** (or Purview Suite / E5 Compliance / E5 eDiscovery add-on) | 1 year |
| All other activity — **non-E5** license | ==180 days== |
| Custom retention (any activity, via retention policy) | Up to 1 year with E5; up to **10 years** with the 10-year retention add-on |

Custom **audit log retention policies** (Purview `Organization Configuration` role required) can extend retention for specific record types/activities/users beyond the defaults, and always take priority over the default policy. Manage them via the Purview portal or PowerShell (`New-UnifiedAuditLogRetentionPolicy`, `Get-UnifiedAuditLogRetentionPolicy`, `Set-UnifiedAuditLogRetentionPolicy`).

> [!warning] Common Mistake
> A question stating a flat "90-day" retention for all Fabric audit activity is describing an **outdated** figure — Audit (Standard)'s default changed from 90 to 180 days for logs generated on or after October 17, 2023. Know the **current** defaults (180 days standard / 1 year E5 / up to 10 years with the add-on), not the pre-2023 numbers.

## Endorsement UI Flow Summary

Regardless of tier, endorsement is applied from the same place — an item's **settings → endorsement section**:

1. Open the item's settings and expand **Endorsement**.
2. Choose **Promoted**, **Certified**, or **Master data** (only options the user is authorized for are selectable; others appear greyed out with a request link).
3. For a Power BI semantic model, an optional **Make discoverable** checkbox additionally lets users without direct access find (but not open) the model in search.
4. Select **Apply**.

An authorized reviewer should treat certification/master-data designation as a genuine review responsibility, not a rubber stamp — Microsoft's own guidance explicitly frames both as actions to take "only if you feel qualified to do so and have reviewed the item."

## Use Cases

- Applying a "Highly Confidential" domain-level default label to all Finance-domain workspaces, distinct from the tenant's general default
- Certifying a small set of gold-standard semantic models so analysts can trust them over ad hoc self-service reports
- Labeling a canonical `dim_Customer` Lakehouse table as Master data so downstream teams know it's the single source of truth
- Auditing who deleted a sensitive Warehouse item, and when, via a `Search-UnifiedAuditLog` query scoped to that item's record type

## Common Issues & Errors

| Issue | Cause | Resolution |
| :--- | :--- | :--- |
| A Power BI report's label didn't propagate to its source Lakehouse | Inheritance doesn't flow Power BI → Fabric | Apply the label directly on the Lakehouse if needed |
| User can't certify an item despite having write permission | Not on the Fabric admin's authorized-reviewers list | Request certification through the org's documented process |
| Domain-level default label isn't applying | The domain-level default sensitivity label feature isn't enabled tenant-wide, or the domain's delegated setting wasn't configured | Confirm the tenant feature is on, then set the label under the domain's Delegated Settings |
| Power BI/Fabric activities missing from audit search results | Auditing wasn't enabled in the Power BI admin portal | Enable **Audit logs** in the Power BI admin portal, then re-run the search |
| Audit search returns nothing beyond 180 days | User's license doesn't qualify for extended retention, and no custom retention policy exists | Assign an E5/Purview Suite license, or create a custom audit log retention policy |
| A restricted admin's search is missing some expected activity | Certain audit activities (AIP Discover, some Exchange/Dynamics 365 operations) are visible only to unrestricted admins | Have an unrestricted admin (no administrative units assigned) run the search instead |
| Capacity ID/name is blank in an audit record | Not all audit records populate this field directly | Cross-reference the activity's timestamp/user against the Microsoft Fabric Capacity Metrics app |
| An item was certified but a scenario says the requester wasn't authorized | Certification requires both write permission **and** tenant-admin authorization as a reviewer | Confirm both conditions — write permission alone is never sufficient for Certified or Master data |

## Best Practices

- Enable domain-level default sensitivity labels only for domains that genuinely need stricter-than-tenant defaults — don't fragment policy unnecessarily
- Reserve Certified/Master data status for genuinely reviewed, trustworthy content — over-certifying erodes the signal's value
- Enable Power BI admin portal auditing early; it's not automatic even with tenant-wide unified audit logging on
- Set up a custom audit log retention policy for compliance-sensitive record types before you need it — retroactive extension isn't possible for already-expired records

## Exam Tips

> [!tip] Exam Tips
>
> - Sensitivity label inheritance: Fabric → Fabric ✅, Fabric → Power BI ✅, Power BI → Fabric ❌ — memorize the one unsupported direction
> - Endorsement tiers: Promoted (any write-permission holder) < Certified (authorized reviewers only) < Master data (authorized reviewers only, data items only)
> - Domain-level default sensitivity labels require a tenant-wide feature flag before a domain admin can set one, and override the tenant default within that domain's scope
> - Fabric audit activity lives in the unified audit log (Purview portal), searchable via UI or `Search-UnifiedAuditLog`; enable Power BI admin portal auditing to actually capture Fabric/Power BI events
> - Default retention: 180 days (non-E5), 1 year (E5), up to 10 years (add-on) — not the older 90-day figure

## Key Takeaways

- Sensitivity labels apply manually, by default policy, or programmatically, and inherit downstream except in the Power BI → Fabric direction
- Domain-level default labels can override the tenant default, but only when the tenant admin has enabled that delegation
- Promoted requires only write permission; Certified and Master data both require tenant-admin authorization on top of write permission
- Fabric activity audits through the unified audit log via Microsoft Purview, with retention tiers driven by license (180 days / 1 year / up to 10 years)
